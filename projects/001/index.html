Exploratory Testing Case Study:
Weather Tracker App
Project Overview & Objectives

Application: Weather Tracker App (Web Application)

Live Test Bed: Live App (You can use this link to replicate the defects documented in Section 3.)

Goal: Fetch and display current weather data based on user-inputted city names, and track recent searches.

Methodology: Exploratory Testing and Session-Based Manual Testing. This approach was chosen to simulate real-world user behavior and uncover non-obvious defects, particularly related to user input validation and state management (recent searches).

Scope: Functional correctness, input validation, display accuracy, and recent city history feature.

Test Design & Strategy

A. Focus Areas (Charters)

Charter Description Technique
Input & Search Test all permutations of user input (valid, invalid, edge cases) and the search button/Enter key functionality. Boundary Value Analysis, Error Guessing.
Data Display Validate that the displayed weather data (temperature, icon, wind, humidity) is present, correctly formatted, and reasonable. Comparison, Data Validation.
Recent History Verify the storage, retrieval, display, and interaction of the "Recent Searches" feature. State Transition Testing, Functional Flow Testing.
Usability & UX Assess the visual clarity, responsiveness, error messaging, and overall flow of the application. Ad-hoc Testing, User Empathy.

B. Environments & Tools

  • Browser: Chrome (Primary Testing) / Firefox (Compatibility Check)
  • DevTools: Used to inspect API call status codes (200, 404) and local storage.
  • Test Data: Real city names, fictional names, numeric strings, empty inputs, international characters.
Identified Defects & Analysis

The following critical and high-priority defects were discovered during the exploratory testing session:

Live Test Bed: Live App (You can use this link to replicate the defects documented in this section.)

ID Severity Feature Defect Description Impact
D-001 High Input Validation Search with just spaces (" Â  ") leads to a successful API call but returns a generic "City not found" error without providing explicit user feedback. Poor user experience and ambiguous error handling.
D-002 Critical Recent History If the user searches for a city using different casing (e.g., "paris" then "Paris"), both entries are saved in the Recent Cities list, duplicating the city name functionality. Data redundancy in local storage; cluttered UI.
D-003 High UI Display Weather icon URL is non-HTTPS (http://...) which causes mixed content warnings or a broken image display in browsers configured for security. Security risk and poor visual fidelity; requires HTTPS endpoint.
D-004 Medium Search Button The search button remains clickable and initiates an API call even when the input field is empty, triggering the "Please enter a city name" error repeatedly. Unnecessary API calls and potential for user confusion.

Key Findings Summary: The core function of fetching weather data is robust. However, the system's data normalization (D-002) and error handling for edge cases (D-001) are weak. The application also contains a minor security/display flaw (D-003) by using an insecure image protocol.

Suggested Improvements & Validation

Based on the defects found, the following remediation steps were recommended and their successful implementation was validated:

Improvement Rationale Validation Check
Input Normalization Before searching or updating recent cities, convert the city name to a standard format (e.g., city.toLowerCase()). Search for "london" then "LONDON"; only one entry appears in Recent Cities. (Fixes D-002)
Trim/Empty Check Add cityInput.value.trim() check at the start of searchCity to handle both empty strings and strings of only spaces. Entering only spaces results in the explicit "Please enter a city name" error, not an API call. (Fixes D-001 & D-004)
API URL Fix Update the iconUrl construction to use the https protocol. Weather icon displays successfully without any console warnings or broken image links. (Fixes D-003)
UX Enhancement Implement input clearing after a successful search to prepare the field for the next query. After searching "Tokyo," the input field is now empty and ready for the next city.
Conclusion

This exploratory testing session demonstrated the application's readiness for core functionality while revealing critical areas for improvement in data integrity, user input handling, and security practices.
