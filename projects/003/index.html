Advanced Test Design & Strategy

The strategy focused on a High-Coverage Contract Testing approach, validating not only the success and failure paths but also critical boundary and unit constraints.

A. Test Scenarios and Coverage

Category Test Case Focus Rationale
Functional/Contract Valid data for a known city (London) Verifies the core data structure (schema) and 200 OK status for the primary successful use case.
Negative Path 404 Not Found for an unknown city Ensures the API correctly signals failure for non-existent resources, allowing the frontend to display the appropriate error message.
Security 401 Unauthorized for an invalid API Key Validates the security mechanism and confirms that authentication is required, preventing unauthorized access.
Boundary Test (Empty) 400/404 for an empty query string Checks how the API handles the minimum valid input (or lack thereof), preventing unexpected server-side errors.
Boundary Test (Long) 400/404 for an excessively long query string Fuzz Testing: Probes the API's input limits to ensure robust behavior and prevent denial-of-service (DoS) potential from oversized requests.
Data Integrity/Unit Data in Celsius when units=metric is unspecified Confirms the units parameter correctly enforces the metric system by asserting the temperature is within a realistic Celsius range (e.g., -50°C to 50°C).

B. Demonstrated Technical Proficiency

The suite utilizes Playwright's APIRequestContext fixture to send HTTP requests and chain assertions using expect() for status codes, JSON properties (toHaveProperty), and data value constraints (toBeGreaterThanOrEqual, toBeLessThanOrEqual).

Playwright Implementation Example

The following code snippet demonstrates how Playwright's test.request fixture is used to execute and validate the core Positive and Negative tests:

/const { test, expect } = require('@playwright/test');

const API_KEY = 'd80899bfa8eecf987dd399fb1d7e1281'; 
const BASE_URL = 'https://api.openweathermap.org/data/2.5';

test.describe('OpenWeatherMap API Contract Tests', () => {

    test('Should return 200 OK and valid data for a known city (London)', async ({ request }) => {
        const city = 'London';
        const response = await request.get(`${BASE_URL}/weather?q=${city}&appid=${API_KEY}&units=metric`);
        
        expect(response.status()).toBe(200);

        const data = await response.json();

        expect(data).toHaveProperty('name', 'London');
        expect(data).toHaveProperty('main');
        expect(data.main).toHaveProperty('temp');
        expect(data.main).toHaveProperty('humidity');
        expect(data).toHaveProperty('weather');
        expect(Array.isArray(data.weather)).toBe(true);
    });

    test('Should return 404 Not Found for an unknown city', async ({ request }) => {
        const city = 'NonExistentCityXYZ123';
        const response = await request.get(`${BASE_URL}/weather?q=${city}&appid=${API_KEY}&units=metric`);
        
        expect(response.status()).toBe(404);

        const data = await response.json();

        expect(data).toHaveProperty('message', 'city not found');
    });

Results, Coverage, and Value Proposition

The automated test suite successfully executed all six API test cases, confirming the stability and compliance of the external service with our application's requirements.

A. Key Coverage Highlights

  • Robust Boundary Handling: Successfully validated that the API gracefully rejects minimum (empty) and maximum (oversized) inputs, preventing potential vulnerabilities or application crashes.
  • Unit of Measure Integrity: Confirmed that the units=metric parameter reliably dictates the Celsius output, ensuring correct data is passed to the UI for display.
  • Decoupled Testing: By testing the API directly, we established a "Shift-Left" QA capability, catching integration issues before the UI is deployed, which significantly reduces the cost and time of finding defects.

B. Value Proposition

  • Speed and Efficiency: These API tests execute in milliseconds, allowing them to run hundreds of times faster than their equivalent UI tests.
  • Reliability: The tests are independent of browser state, rendering, and asynchronous UI events, making them extremely reliable (non-flaky).
Conclusion

This API automation project demonstrates proficiency in building fast, comprehensive, and maintenance-friendly test suites for external dependencies. It provides a foundational safety net for the Weather Tracker App's core functionality.
